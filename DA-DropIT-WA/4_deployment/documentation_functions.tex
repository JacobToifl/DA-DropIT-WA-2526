\section{Funktionen}

Dieses Kapitel gibt einen Überblick über die zentralen Funktionen des umgesetzten Systems. Dabei werden insbesondere der Dokumenten-Upload, die Dokumenten-Klassifikation sowie die Dokumenten-Suche und deren technischer Ablauf beschrieben.

\subsection{Dokumenten-Upload}\label{subsec:dokumenten_upload}

Dieses Kapitel dokumentiert den vollständigen Upload-Flow von der Anlage einer Upload-Zone bis zur finalen Speicherung des Dokuments in SharePoint inklusive Metadatenpersistenz und Suchindex-Aktualisierung.

\subsubsection{Technischer Gesamtflow}

Der Upload ist als mehrstufige Pipeline umgesetzt, damit fachliche Prüfung, Benutzerinteraktion und Persistenz klar getrennt bleiben:
\begin{enumerate}
    \item Eine Zone wird im Frontend erstellt oder ausgewählt.
    \item Die Zone wird im Zone-Service persistiert; der zugehörige SharePoint-Ordner wird best-effort angelegt.
    \item Eine Datei wird per Dateiauswahl oder Drag \& Drop in die Zone-Upload-Queue gestellt.
    \item Der Zone-Service führt eine Preflight-Phase aus und ruft die Dokumenten-Klassifikation auf.
    \item Ergebniswerte der Klassifikation werden zusammen mit der Datei temporär abgelegt und mit \texttt{uploadId} referenziert.
    \item Die Benutzerin bzw. der Benutzer bestätigt oder korrigiert die erkannten Metadaten.
    \item Beim \texttt{commit} wird die Datei nach SharePoint geschrieben und die temporäre Ablage aufgelöst.
    \item Die Dokument-Metadaten werden in der Documents-API in der Datenbank gespeichert.
    \item Das Dokument wird für die Suche über die Search-API indiziert.
\end{enumerate}

Damit kann der Uploadprozess robust aufgeteilt werden: Die KI-Extraktion ist vor der finalen Persistenz sichtbar und korrigierbar, bevor irreversible Schreiboperationen durchgeführt werden.

\subsubsection{Zone-Erstellung und SharePoint-Ordnerinitialisierung}

Der Upload beginnt fachlich mit einer Zone. Die Zone-API stellt dafür \texttt{POST /zones} bereit. Beim Anlegen wird ein neuer \texttt{ZoneDto}-Eintrag in der zonenspezifischen JSON-Struktur gespeichert und parallel versucht, den Zielordner in SharePoint zu erzeugen.

\begin{lstlisting}[language={[Sharp]C},caption={Zone-Erstellung mit SharePoint-Ordnerinitialisierung},label={lst:upload-zone-create}]
g.MapPost(
    "",
    async (
        ZoneDto zone,
        IZonesStore store,
        IUserContext user,
        ITokenAcquisition tokenAcquisition,
        IHttpClientFactory httpFactory,
        IConfiguration cfg,
        CancellationToken ct
    ) =>
{
    var folderName = zone.FolderPath.Trim('/').Split('/').Last();
    var driveId = cfg["SharePoint:DriveId"] ?? zone.DriveId ?? "";

    var doc = await store.UpsertAsync(
        user.TenantId,
        user.ObjectId,
        d =>
        {
            var list = d.Zones.ToList();
            var newZone = zone with
            {
                Id = Guid.NewGuid(),
                OrderIndex = list.Count + 1,
                DriveId = driveId
            };

            list.Add(newZone);
            return d with { Zones = list };
        },
        null,
        ct
    );

    try
    {
        var token = await tokenAcquisition.GetAccessTokenForUserAsync(
            new[] { "https://graph.microsoft.com/Files.ReadWrite.All" });

        var client = httpFactory.CreateClient("graph");
        client.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", token);

        var payload = new
        {
            name = folderName,
            folder = new { },
            microsoft_graph_conflictBehavior = "fail"
        };

        await client.PostAsJsonAsync($"drives/{driveId}/root/children", payload, ct);
    }
    catch { }

    return Results.Ok(doc);
}
);
\end{lstlisting}

Die Ordneranlage ist absichtlich tolerant implementiert (\texttt{try/catch} ohne Abbruch), sodass die fachliche Zone nicht verloren geht, wenn die Graph-Operation nicht unmittelbar erfolgreich ist.

\subsubsection{Frontend-Uploadsteuerung über UploadQueueService}

In der Webanwendung wird der Upload nicht direkt aus der Seite abgesendet, sondern über den \texttt{UploadQueueService} sequenziell verarbeitet. Dadurch können mehrere Dateien nacheinander pro Zone verarbeitet werden, inklusive Statusmeldungen und Dialogsteuerung.

\begin{lstlisting}[language={[Sharp]C},caption={Preflight- und Commit-Steuerung im UploadQueueService},label={lst:upload-queue-preflight-commit}]
private async Task ProcessRequestAsync(UploadRequest request, CancellationToken ct)
{
    if (request.SizeBytes > MaxUploadBytes) return; // 100 MB Grenze

    using var form = new MultipartFormDataContent();
    form.Add(new StreamContent(await OpenUploadStreamAsync(request, ct)),
             "file", request.FileName);

    var res = await client.PostAsync($"zones/{request.ZoneId}/upload/preflight", form, ct);
    if (!res.IsSuccessStatusCode) return;

    var preflight = await res.Content.ReadFromJsonAsync<UploadPreflight>(cancellationToken: ct);
    CurrentPreflight = preflight;
    _decision = new TaskCompletionSource<UploadDecision>(TaskCreationOptions.RunContinuationsAsynchronously);

    var decision = await _decision.Task.WaitAsync(ct);
    if (!decision.Confirmed || decision.Payload is null) return;

    await CommitAsync(client, decision.Payload, ct);
}
\end{lstlisting}

Wesentliche technische Punkte in dieser Schicht sind:
\begin{itemize}
    \item zentrale Dateigrößenbegrenzung auf 100 MB,
    \item einheitlicher Status pro Zone (z. B. \texttt{"KI analysiert Dokument..."}),
    \item ein expliziter Bestätigungszustand zwischen Preflight und Commit,
    \item sowie identischer Ablauf für Dateiauswahl und Drag-\&-Drop.
\end{itemize}

\subsubsection{Preflight-Endpunkt und Klassifikationsaufruf}

Die eigentliche serverseitige Vorverarbeitung startet über den Preflight-Endpunkt der Zone-API. Der Endpunkt nimmt das Multipart-File entgegen, sendet es an den AI-Service und legt danach Datei plus Metadaten temporär ab.

\begin{lstlisting}[language={[Sharp]C},caption={Preflight-Endpunkt im Zone-Service},label={lst:upload-preflight-endpoint}]
g.MapPost(
    "{zoneId:guid}/upload/preflight",
    async (
        Guid zoneId,
        HttpRequest req,
        FileZonesStore store,
        IUserContext user,
        IHttpClientFactory httpFactory,
        CancellationToken ct
    ) =>
{
    var form = await req.ReadFormAsync(ct);
    var file = form.Files["file"];
    if (file == null) return Results.BadRequest("No file");

    var ai = httpFactory.CreateClient("ai");
    using var mp = new MultipartFormDataContent();
    mp.Add(new StreamContent(file.OpenReadStream()), "file", file.FileName);

    var aiResp = await ai.PostAsync("/ai/classify", mp, ct);
    if (!aiResp.IsSuccessStatusCode) return Results.Problem("KI-Dienst Fehler");

    var extraction =
        await aiResp.Content.ReadFromJsonAsync<DocumentClassificationResultDto>(
            new JsonSerializerOptions
            {
                PropertyNamingPolicy = JsonNamingPolicy.CamelCase
            },
            ct
        );

    var (uploadId, docType) = await store.SaveTempAsync(
        user.TenantId,
        user.ObjectId,
        zoneId,
        file,
        extraction!.DocumentType,
        extraction.DocumentYear,
        extraction.ContractType,
        extraction.Partner,
        extraction.ExpireDate,
        ct
    );

    return Results.Ok(new
    {
        uploadId,
        documentType = docType.ToString(),
        year = extraction.DocumentYear,
        partner = extraction.Partner,
        contractType = extraction.ContractType,
        expireDate = extraction.ExpireDate
    });
}
).DisableAntiforgery();
\end{lstlisting}

Die Klassifikation selbst (Prompting, Schema-Bindung, Nachvalidierung) ist in Kapitel 4.2.2 detailliert beschrieben und wird hier nur als aufgerufener Baustein verwendet.

Die Methode \texttt{SaveTempAsync(...)} persistiert in dieser Phase:
\begin{itemize}
    \item die hochgeladene Binärdatei im temporären Bereich,
    \item ein Metadaten-JSON mit \texttt{uploadId}, Zone, Dateieigenschaften und Klassifikation,
    \item sowie den Zeitstempel der KI-Auswertung (\texttt{ClassifiedAtUtc}).
\end{itemize}

\subsubsection{Temporäre Ablage und Benutzerkorrektur}

Die temporäre Ablage entkoppelt KI-Ergebnis und finale Speicherung. Das Frontend zeigt die zurückgegebenen Werte in einem Dialog (\texttt{UploadPreflight}) an. Dort können Dokumenttyp, Jahr, Partner, Vertragsart und Ablaufdatum vor dem Commit angepasst werden.

Die finale Commit-Nachricht enthält damit nicht nur die \texttt{uploadId}, sondern den bestätigten Metadatenstand aus der Benutzeroberfläche.

\subsubsection{Commit in SharePoint und technische Verarbeitung}

Der Commit wird über \texttt{POST /zones/\{zoneId\}/upload/\{uploadId\}/commit} gestartet. Die Hauptarbeit liegt in \texttt{CommitTempToSharePointAsync(...)}:
\begin{itemize}
    \item Laden der temporären Datei und KI-Metadaten,
    \item Merge mit benutzerbestätigten Änderungen,
    \item SHA-256-Bildung zur Integritätsreferenz,
    \item Zielpfad-Aufbau aus Zone-Ordner und optionalem Partner-Unterordner,
    \item Upload der Datei in SharePoint via Microsoft Graph,
    \item optionale Textextraktion bei PDF für spätere Suche,
    \item Löschung der temporären Artefakte.
\end{itemize}

\begin{lstlisting}[language={[Sharp]C},caption={Commit-Logik mit SharePoint-Upload},label={lst:upload-commit-store}]
var finalMeta = aiMeta with
{
    ExpireDate = confirmed.ExpireDate,
    Partner = confirmed.Partner ?? aiMeta.Partner,
    ContractType = confirmed.ContractType ?? aiMeta.ContractType,
    DocumentYear = confirmed.DocumentYear ?? aiMeta.DocumentYear,
    DocumentType = confirmed.DocumentType ?? aiMeta.DocumentType
};

var partnerFolder = SanitizeFolderName(finalMeta.Partner);
if (!string.IsNullOrWhiteSpace(partnerFolder))
{
    await EnsureFolderExistsAsync(client, driveId, zoneFolder, partnerFolder, ct);
    targetFolder = string.IsNullOrWhiteSpace(zoneFolder) ? partnerFolder : $"{zoneFolder}/{partnerFolder}";
}

var uploadUrl = string.IsNullOrEmpty(targetFolder)
    ? $"drives/{encodedDriveId}/root:/{encodedFileName}:/content"
    : $"drives/{encodedDriveId}/root:/" +
      $"{EncodePathForGraph(targetFolder)}/{encodedFileName}:/content";

using (var stream = File.OpenRead(binPath))
{
    var resp = await client.PutAsync(uploadUrl, new StreamContent(stream), ct);
    resp.EnsureSuccessStatusCode();
    var item = await resp.Content.ReadFromJsonAsync<JsonElement>(ct);
    sharePointItemId = item.GetProperty("id").GetString()!;
}
\end{lstlisting}

Die Partner-Unterordner werden vor dem Datei-Upload gezielt erstellt. Ungültige Zeichen werden über \texttt{SanitizeFolderName(...)} ersetzt, damit SharePoint-konforme Pfade entstehen.

\subsubsection{Persistenz in Documents-API und Suchindex}

Nach erfolgreichem SharePoint-Write registriert der Zone-Service das Dokument in der Documents-API und übergibt dabei auch \texttt{sharePointItemId} und \texttt{ownerId}. Danach wird ein interner Indexierungsaufruf an die Search-API ausgelöst.

var (meta, storedPath, sha256, contentText, sharePointItemId) =

var dto = new CreateDocumentDto(

var regResp = await docApi.PostAsJsonAsync("/documents", dto, ct);
var created = await regResp.Content.ReadFromJsonAsync<CreatedDocumentResponse>(ct);

searchApi.DefaultRequestHeaders.TryAddWithoutValidation(

Die Suche wird damit direkt nach dem Upload aktualisiert; es ist kein separater Batchlauf notwendig, um neue Dokumente auffindbar zu machen.

\subsubsection{Fehlerbehandlung und Sicherheitsaspekte im Upload-Flow}

Der Flow implementiert mehrere Sicherheits- und Robustheitsmechanismen:
\begin{itemize}
    \item Autorisierung auf allen geschützten Zone-Endpunkten über Benutzerkontext (\texttt{TenantId}, \texttt{ObjectId}),
    \item temporäre Entkopplung via \texttt{uploadId}, um unbestätigte Uploads nicht sofort final zu persistieren,
    \item typisierte Persistenzobjekte für API-Grenzen (DTOs für Dokumenterfassung und Suchindexierung),
    \item interne Indexierung nur mit \texttt{X-Internal-Key},
    \item ACL-Übernahme in den Index über \texttt{AllowedUsers}.
\end{itemize}

Zusätzlich führen Fehler in Preflight, Commit oder Metadatenpersistenz zu klaren Benutzer-Rückmeldungen im Frontend (Toasts) und unterbrechen den Vorgang kontrolliert, statt inkonsistente Zwischenzustände zu erzeugen.

\subsubsection{Zusammenfassung der Upload-Implementierung}

Die Upload-Implementierung verbindet Benutzerführung, KI-Voranalyse und Persistenz in einem durchgängigen Prozess.
Durch die Trennung in Preflight und Commit bleiben KI-Ergebnisse vor dem endgültigen Speichern überprüfbar. Datenhaltung und Suchindexierung werden danach konsistent aktualisiert.

\subsection{Dokumenten-Klassifikation}\label{subsec:dokumenten_klassifikation}

Dieses Kapitel dokumentiert die implementierte Dokumenten-Klassifikation und beschreibt den technischen Ablauf vom API-Endpunkt bis zur Nachvalidierung der Ergebnisse.

\subsubsection{Zweck und Ergebnisobjekt}

Die Klassifikation extrahiert aus einem PDF die für die weitere Verarbeitung relevanten Metadaten und liefert diese in einem einheitlichen DTO zurück. Das Ergebnis besteht aus Dokumenttyp, Jahr, Vertragstyp, Partner und optionalem Ablaufdatum.

\begin{lstlisting}[language={[Sharp]C},caption={Rückgabeobjekt der Klassifikation},label={lst:classification-result-dto}]
public record DocumentClassificationResultDto(
    DocumentType DocumentType,
    int? DocumentYear,
    string? ContractType,
    string? Partner,
    DateTime? ExpireDate
);
\end{lstlisting}

Damit ist das Ergebnisformat für alle aufrufenden Komponenten stabil und typisiert.

\subsubsection{Klassifikations-Endpunkt}

Die AI-API stellt den Endpunkt \texttt{POST /ai/classify} bereit. Der Controller validiert die Eingabe (nicht leer, MIME-Type PDF) und delegiert die eigentliche Klassifikationslogik an den Service \texttt{OpenAiClassifier}.

\begin{lstlisting}[language={[Sharp]C},caption={API-Endpunkt der Dokumenten-Klassifikation},label={lst:classification-endpoint}]
[HttpPost("classify")]
public async Task<ActionResult<DocumentClassificationResultDto>> Classify(
    [FromForm] IFormFile file, CancellationToken ct)
{
    if (file.Length == 0) return BadRequest("Empty file.");
    if (!string.Equals(file.ContentType, "application/pdf",
        StringComparison.OrdinalIgnoreCase))
        return BadRequest("Only PDF supported for now.");

    await using var s = file.OpenReadStream();
    var result = await _classifier.ClassifyPdfAsync(s, file.FileName, ct);
    return Ok(result);
}
\end{lstlisting}

Die Verantwortung des Controllers bleibt damit bewusst schlank: Validierung, Delegation und Response.

\subsubsection{Kernlogik im OpenAiClassifier}

Der Service \texttt{OpenAiClassifier} enthält die eigentliche KI-Interaktion. Die Pipeline besteht aus vier Schritten:
\begin{enumerate}
    \item Upload der PDF-Datei an die OpenAI Files API,
    \item Aufruf der Responses API mit Modell, Prompt und strikt vorgegebenem JSON-Schema,
    \item Parsen der strukturierten Antwort,
    \item technische und fachliche Nachvalidierung der Felder.
\end{enumerate}

Der wichtigste Baustein ist das Schema-gebundene Prompting, weil dadurch die Antwort in ein verlässliches, maschinenlesbares Format gezwungen wird.

\begin{lstlisting}[language={[Sharp]C},caption={Schema-gebundener Responses-Aufruf},label={lst:classification-schema-call}]
var payload = new
{
    model = _model,
    input = new[]
    {
        new {
            role = "user",
            content = new object[]
            {
                new { type = "input_file", file_id = fileId },
                new { type = "input_text", text = prompt }
            }
        }
    },
    text = new
    {
        format = new
        {
            type = "json_schema",
            name = "doc_extraction",
            strict = true,
            schema
        }
    }
};

using var resp = await _http.PostAsync(
    "https://api.openai.com/v1/responses",
    new StringContent(
        JsonSerializer.Serialize(payload),
        Encoding.UTF8,
        "application/json"
    ),
    ct
);
\end{lstlisting}

Durch \texttt{strict = true} und das definierte Schema wird die Ausgabe stark normalisiert. Das reduziert nachgelagerte Parsingfehler.

\subsubsection{Parsing und fachliche Nachvalidierung}

Nach dem API-Aufruf wird der JSON-Text extrahiert, in die internen Typen gemappt und anschließend fachlich bereinigt. Dabei gelten unter anderem folgende Regeln:
\begin{itemize}
    \item Jahr nur im Bereich 1900 bis 2100,
    \item \texttt{ContractType} und \texttt{ExpireDate} nur für \texttt{DocumentType.Vertrag},
    \item leere Strings werden auf \texttt{null} normalisiert.
\end{itemize}

\begin{lstlisting}[language={[Sharp]C},caption={Nachvalidierung der Klassifikationsergebnisse},label={lst:classification-post-validation}]
var typeStr = root.GetProperty("document_type").GetString()!;
var docType = Enum.Parse<DocumentType>(typeStr, ignoreCase: true);

int? year = null;
var y = root.GetProperty("year");
if (y.ValueKind == JsonValueKind.Number && y.TryGetInt32(out var yi))
    year = yi;

if (year is < 1900 or > 2100) year = null;
if (docType != DocumentType.Vertrag)
{
    contractType = null;
    expireDate = null;
}
\end{lstlisting}

Diese Nachvalidierung ist zentral, damit inkonsistente Modellantworten nicht ungeprüft in den weiteren Prozess übernommen werden.

\subsubsection{Fehlerbehandlung und Robustheit}

Die Service-Methoden prüfen HTTP-Antworten zentral über \texttt{EnsureOk(...)}. Bei nicht erfolgreichem Status wird die API-Antwort inklusive Statuscode in eine Exception übernommen. Dadurch sind Fehler in Files-Upload oder Responses-Aufruf klar lokalisierbar und beobachtbar.

Zusätzlich unterstützt \texttt{ExtractFirstOutputText(...)} mehrere Antwortvarianten, insbesondere ein direktes \texttt{output\_text}-Feld sowie textuelle Inhalte in der \texttt{output}-Struktur. Das macht die Verarbeitung robuster gegenüber Response-Formvarianten.

\subsubsection{Zusammenfassung der Klassifikationsimplementierung}

Die Dokumenten-Klassifikation ist als klar gekapselter Service umgesetzt:
\begin{itemize}
    \item schlanker API-Endpunkt für PDF-Eingaben,
    \item schema-basierter KI-Aufruf für strukturierte Extraktion,
    \item typisierte Ergebnisrückgabe über \texttt{DocumentClassificationResultDto},
    \item sowie explizite Nachvalidierung der KI-Ergebnisse.
\end{itemize}

Damit ist die Klassifikationslogik technisch isoliert dokumentiert und als wiederverwendeter Baustein beschrieben.

\subsection{Dokumenten-Suche}

Dieses Kapitel dokumentiert die implementierte Suchfunktion über den gesamten Request-Flow hinweg: von der Benutzerinteraktion in der Weboberfläche bis zur Ausführung der Abfrage in Azure AI Search und der Rückgabe von Treffern inklusive Facetten.

\subsubsection{Technischer Such-Flow}

Die Suche ist als mehrstufiger Ablauf umgesetzt:
\begin{enumerate}
    \item Die Benutzerin bzw. der Benutzer gibt Suchtext und optionale Filter (Dokumenttyp, Vertragstyp, Jahr, Partner) in der Seite \texttt{Search.razor} ein.
    \item Das Web-Frontend baut daraus ein \texttt{SearchRequestDto} und sendet einen \texttt{POST} an den Search-Service (\texttt{/search}).
    \item Der Search-Service liest die Benutzeridentität (\texttt{oid}) aus dem Access Token und erzwingt damit eine benutzerbezogene Zugriffskontrolle.
    \item Der \texttt{AzureSearchQueryService} erstellt \texttt{SearchOptions} (Paging, Facets, Filter) und führt die Abfrage gegen den Azure-Search-Index aus.
    \item Die Treffer und Facetten werden in ein \texttt{SearchResponseDto} gemappt und an das Frontend zurückgegeben.
    \item Das Frontend aktualisiert Trefferliste und Filteroptionen dynamisch anhand der zurückgegebenen Facetten.
\end{enumerate}

\subsubsection{Frontend: Aufbau und Versand des Such-Requests}

Die zentrale Suchmethode im Frontend befindet sich in \texttt{Search.razor}. Vor dem eigentlichen Suchaufruf wird eine Synchronisation mit der Zone-API ausgelöst. Anschließend wird das DTO aufgebaut und an den Suchendpunkt gesendet.

\begin{lstlisting}[language={[Sharp]C},caption={Suchen im Frontend mit DTO-Aufbau und API-Aufruf},label={lst:search-frontend-request}]
private async Task RunSearch()
{
    await ZoneApi.PostAsync("/zones/sync-all", null);

    var req = new SearchRequestDto(
        Query: string.IsNullOrWhiteSpace(_query) ? null : _query.Trim(),
        DocumentYear: int.TryParse(_documentYear, out var y) ? y : null,
        DocumentType:
            Enum.TryParse<DocumentType>(_documentType, true, out var dt)
                ? dt
                : null,
        ContractType: string.IsNullOrWhiteSpace(_contractType) ? null : _contractType,
        Partner: string.IsNullOrWhiteSpace(_partner) ? null : _partner,
        Skip: 0,
        Size: 50
    );

    var resp = await SearchApi.PostAsJsonAsync("/search", req);
    resp.EnsureSuccessStatusCode();
}
\end{lstlisting}

Die Implementierung ist bewusst defensiv: Leere Eingaben werden als \texttt{null} übertragen, numerische und Enum-Filter werden nur bei erfolgreicher Konvertierung gesetzt. Dadurch bleiben Request und Filterlogik stabil, auch wenn einzelne UI-Felder leer bleiben.

\subsubsection{Suchmodi und Filterkombinationen}

Die Suche unterstützt mehrere praktische Nutzungsarten:
\begin{itemize}
    \item Volltextsuche über \texttt{Query} (z. B. Begriff aus Dateiname oder Dokumentinhalt),
    \item reine Filterabfrage ohne Suchtext (z. B. nur Jahr, Partner oder Dokumenttyp),
    \item kombinierte Suche aus Volltext und Filtern (z. B. Suchwort plus Jahr und Partner),
    \item gezielte Eingrenzung über Vertragsart (\texttt{ContractType}) bei Vertragsdokumenten.
\end{itemize}

Technisch wird bei leerem Suchtext automatisch \texttt{"*"} als Suchausdruck verwendet. Dadurch kann das System auch dann Ergebnisse liefern, wenn ausschließlich nach Metadaten wie Jahr, Partner oder Dokumenttyp gefiltert wird.

\subsubsection{Search API: Authentifizierter Endpunkt}

Der Search-Service stellt die fachliche Suchoperation über \texttt{POST /search} bereit. Der Endpunkt ist geschützt (\texttt{RequireAuthorization()}) und extrahiert die Benutzerkennung aus dem JWT-Token. Ohne gültige \texttt{oid} wird der Request abgelehnt.

\begin{lstlisting}[language={[Sharp]C},caption={Search-Endpunkt mit Benutzerkontext},label={lst:search-api-endpoint}]
app.MapPost(
    "/search",
    async (
        SearchRequestDto req,
        HttpContext ctx,
        AzureSearchQueryService svc,
        CancellationToken ct
    ) =>
{
    var oid = ctx.User.FindFirst("oid")?.Value
              ?? ctx.User.FindFirst(
                  "http://schemas.microsoft.com/identity/claims/objectidentifier"
                 )?.Value;

    if (string.IsNullOrWhiteSpace(oid)) return Results.Unauthorized();

    var result = await svc.SearchAsync(req, oid, ct);
    return Results.Ok(result);
}
).RequireAuthorization();
\end{lstlisting}

Damit ist sichergestellt, dass jeder Suchaufruf mit einem konkreten Benutzerkontext verknüpft ist und die nachgelagerte Filterlogik diesen Kontext verwenden kann.

\subsubsection{Query-Service: Paging, Facets und Filter}

Die eigentliche Suchlogik liegt im \texttt{AzureSearchQueryService}. Dort werden Suchtext, Paging und Facetten konfiguriert. Für leere Suchanfragen wird ein Wildcard-Searchtext (\texttt{"*"}) verwendet, damit auch reine Filterabfragen möglich sind.

\begin{lstlisting}[language={[Sharp]C},caption={SearchOptions und Filteraufbau im Query-Service},label={lst:search-query-options}]
var searchText = string.IsNullOrWhiteSpace(req.Query) ? "*" : req.Query;

var opt = new SearchOptions
{
    IncludeTotalCount = true,
    Skip = Math.Max(0, req.Skip),
    Size = Math.Clamp(req.Size, 1, 100)
};

opt.Facets.Add("documentYear,count:50");
opt.Facets.Add("documentType,count:50");
opt.Facets.Add("partner,count:50");

opt.Filter = BuildFilter(
    req,
    callerOid,
    includeDocumentType: true,
    includeDocumentYear: true,
    includeContractType: true,
    includePartner: true
);

var resp = await _client.SearchAsync<SearchDocument>(searchText, opt, ct);
\end{lstlisting}

Die Limits (\texttt{Size} zwischen 1 und 100) schützen den Service vor übergroßen Antworten. Gleichzeitig liefern Facetten die Datenbasis für dynamische Filteroptionen im UI.

Die wichtigste Sicherheitsregel ist im Filteraufbau implementiert: Ergebnisse werden immer auf Dokumente eingeschränkt, die für den aktuellen Benutzer freigegeben sind.

\begin{lstlisting}[language={[Sharp]C},caption={Benutzerbezogener OData-Filter},label={lst:search-filter-acl}]
private static string BuildFilter(SearchRequestDto req, string callerOid, ...)
{
    var parts = new List<string>();

    parts.Add($"allowedUsers/any(u: u eq '{EscapeOData(callerOid)}')");

    if (req.DocumentYear is not null)
        parts.Add($"documentYear eq {req.DocumentYear.Value}");
    if (req.DocumentType is not null)
        parts.Add($"documentType eq '{EscapeOData(req.DocumentType.Value.ToString())}'");
    if (!string.IsNullOrWhiteSpace(req.Partner))
        parts.Add($"partner eq '{EscapeOData(req.Partner)}'");

    return string.Join(" and ", parts);
}
\end{lstlisting}

Durch die Kombination aus ACL-Bedingung und fachlichen Filtern entsteht ein Security-First-Ansatz: Ein Dokument wird nur dann zurückgegeben, wenn sowohl die Berechtigung als auch die Suchkriterien erfüllt sind.

\subsubsection{Rückgabe und UI-Aktualisierung}

Nach der Suchausführung werden die Index-Dokumente auf \texttt{SearchHitDto}-Objekte gemappt. Zusätzlich werden Facettenwerte in Bucket-Strukturen überführt. Das Frontend nutzt diese Buckets, um Filterlisten (z. B. Jahre, Vertragstypen, Partner) mit Trefferanzahl zu aktualisieren.

Dadurch entsteht eine adaptive Suchoberfläche: Filter zeigen nur Werte an, die im aktuellen Ergebnisraum tatsächlich vorkommen. Das reduziert ungültige Kombinationen und verbessert die Bedienbarkeit bei großen Dokumentmengen.

\subsubsection{Zusammenfassung der Implementierung}

Die implementierte Dokumentensuche kombiniert:
\begin{itemize}
    \item einen authentifizierten API-Endpunkt,
    \item benutzerbezogene Zugriffskontrolle direkt in der Query,
    \item kombinierbare Volltext- und Metadatenfilter,
    \item serverseitige Facettenaggregation,
    \item sowie eine dynamische UI-Nachführung der Filteroptionen.
\end{itemize}

Damit ist die Suche sowohl funktional als auch sicherheitstechnisch in die Gesamtarchitektur integriert und für den produktiven Einsatz mit mandanten-/benutzerbezogenen Daten geeignet.

\subsection{Benutzer- und Rollenverwaltung}\label{subsec:benutzer_und_rollenverwaltung}

Die Benutzer- und Rollenverwaltung basiert auf Microsoft Entra ID. Eine lokale
Benutzerverwaltung mit eigenen Passwörtern ist in der aktuellen Lösung nicht
vorgesehen. Stattdessen übernimmt Entra ID die Anmeldung, stellt die
Identitäts-Claims bereit und liefert damit die Grundlage für geschützte
UI-Bereiche sowie autorisierte API-Aufrufe.

\subsubsection{Authentifizierter Zugriff auf Webanwendung und Dienste}

Die Webanwendung ist so konfiguriert, dass die Anmeldung über Microsoft Entra ID
erfolgt und nach erfolgreicher Authentifizierung Zugriffstoken für
nachgelagerte APIs angefordert werden können. Gleichzeitig werden die zentralen
UI-Komponenten nur für angemeldete Benutzer gerendert. Auf Serverseite
validieren insbesondere Zone- und Search-Service die mitgesendeten JWT-Tokens
und schützen ihre Endpunkte über \texttt{RequireAuthorization()}.

\begin{lstlisting}[language={[Sharp]C},caption={Authentifizierte Webanwendung mit geschütztem UI-Zugriff},label={lst:user-auth-web}]
builder.Services
    .AddAuthentication(OpenIdConnectDefaults.AuthenticationScheme)
    .AddMicrosoftIdentityWebApp(builder.Configuration.GetSection("AzureAd"))
    .EnableTokenAcquisitionToCallDownstreamApi()
    .AddInMemoryTokenCaches();

app.MapRazorComponents<App>()
    .AddInteractiveServerRenderMode()
    .RequireAuthorization();
\end{lstlisting}

Durch diese Kopplung von Anmeldung und Tokenbeschaffung fungiert die
Webanwendung als zentraler Einstiegspunkt für den Benutzer, während die
Fachdienste getrennt und zielgerichtet angesprochen werden können.

\subsubsection{Benutzerkontext aus Entra-Claims}

Für die fachliche Verarbeitung wird aus den Entra-Claims ein technischer
Benutzerkontext gebildet. Relevant sind vor allem Tenant-ID und Object-ID, da
diese Identifikatoren im weiteren Verlauf für benutzerbezogene Datenzugriffe
verwendet werden. Zusätzlich werden Anzeigename und E-Mail aus den Claims
übernommen.

\begin{lstlisting}[language={[Sharp]C},caption={Ermittlung des Benutzerkontexts aus Entra-Claims},label={lst:user-context-claims}]
public string TenantId =>
    U.FindFirstValue("tid") ??
    U.FindFirst("http://schemas.microsoft.com/identity/claims/tenantid")?.Value ??
    "";

public string ObjectId =>
    U.FindFirstValue("oid") ??
    U.FindFirst("http://schemas.microsoft.com/identity/claims/objectidentifier")?.Value ??
    U.FindFirstValue(ClaimTypes.NameIdentifier) ??
    "";
\end{lstlisting}

In der Zone-API wird dieser Kontext beispielsweise verwendet, um Zonen pro
Mandant und Benutzer zu laden oder Dateien einem konkreten Eigentümer
zuzuordnen. Auch die Suche greift auf die übermittelte \texttt{oid} zurück, um
nur jene Dokumente zurückzugeben, die für den aktuellen Benutzer freigegeben
sind.

\subsubsection{Rollenauflösung im Frontend}

Neben der reinen Benutzeridentität werden auch Rollen aus dem Anmeldekontext
ausgewertet. Dazu liest der \texttt{CurrentUserService} die Claims
\texttt{roles} und \texttt{ClaimTypes.Role} aus. Falls im Token keine explizite
Rolle enthalten ist, wird standardmäßig die Rolle \texttt{Mitarbeiter}
verwendet.

\begin{lstlisting}[language={[Sharp]C},caption={Rollenauflösung mit Standardrolle},label={lst:user-roles-default}]
public async Task<IReadOnlyList<string>> GetRolesAsync()
{
    var user = await GetUserAsync();
    var roles = new List<string>();

    if (user.Identity?.IsAuthenticated == true)
    {
        roles.AddRange(user.FindAll("roles").Select(c => c.Value));
        roles.AddRange(user.FindAll(ClaimTypes.Role).Select(c => c.Value));
    }

    if (roles.Count == 0)
    {
        roles.Add(DefaultRole);
    }

    return roles.Distinct().ToList();
}
\end{lstlisting}

Die ermittelten Rollen werden im Frontend derzeit vor allem für die
Benutzeranzeige verwendet, etwa im Benutzer-Popover der Navigationsleiste. Eine
umfangreiche fachliche Differenzierung mehrerer Rollen ist in der aktuellen
Ausbaustufe jedoch nur vorbereitet und noch nicht als durchgängige
Berechtigungslogik umgesetzt.

\subsubsection{Benutzerbezogene Zugriffskontrolle in den Diensten}

Die wirksame Zugriffskontrolle erfolgt in der aktuellen Implementierung primär
benutzerbezogen. Geschützte Endpunkte in der Zone- und Search-API verlangen eine
gültige Authentifizierung, während die konkrete Datenfreigabe über die
Entra-Identität des aufrufenden Benutzers erfolgt. So werden Suchtreffer in der
Search-API anhand der \texttt{oid} eingeschränkt, und in der Documents-API wird
die Eigentümerzuordnung über die übergebene Benutzer-ID
(\texttt{X-User-Oid}) abgesichert.

Diese Kombination aus geschütztem Zugriff, Claim-basierter
Benutzeridentifikation und vorbereiteter Rollenauflösung schafft eine solide
Grundlage für ein erweiterbares Berechtigungssystem. Gleichzeitig bleibt die
aktuelle Lösung technisch überschaubar, da die operative Freigabelogik noch
nicht auf komplexe Rollenmatrizen angewiesen ist.

\subsubsection{Zusammenfassung der Implementierung}

Die implementierte Benutzer- und Rollenverwaltung kombiniert:
\begin{itemize}
    \item zentrale Anmeldung über Microsoft Entra ID,
    \item Token-basierte Absicherung von Webanwendung und APIs,
    \item Extraktion von Benutzeridentitäten aus Claims,
    \item eine einfache Rollenauflösung mit Standardrolle,
    \item sowie benutzerbezogene Freigaben in Suche, Zonen- und Dokumentdiensten.
\end{itemize}

Damit ist die Benutzer- und Rollenverwaltung bereits in die technische
Gesamtarchitektur integriert und bietet eine belastbare Basis für spätere,
feingranularere Rollen- und Berechtigungserweiterungen.

\subsection{API-Endpunkte}\label{subsec:api_endpunkte}

Dieses Unterkapitel gibt einen kompakten Überblick über die wichtigsten
API-Endpunkte der einzelnen Services. Im Fokus stehen dabei jene Endpunkte, die
im tatsächlichen Ablauf von Upload, Dokumentverwaltung, Suche und
Benutzerzugriff verwendet werden.

\subsubsection{Zone-Service}

Der Zone-Service stellt die zentrale fachliche API des Systems bereit. Er
verwaltet Upload-Zonen, steuert den Upload-Prozess und bildet die Brücke zu
SharePoint sowie zu den nachgelagerten Dokument- und Suchdiensten.

\begin{itemize}
    \item \texttt{GET /zones}: Lädt alle Zonen des aktuell angemeldeten Benutzers.
    \item \texttt{POST /zones}: Legt eine neue Zone an und initialisiert den zugehörigen SharePoint-Ordner.
    \item \texttt{PUT /zones/\{id\}}: Aktualisiert Eigenschaften einer bestehenden Zone, z. B. Name oder Aktiv-Status.
    \item \texttt{DELETE /zones/\{id\}}: Entfernt eine Zone aus der benutzerbezogenen Zonenkonfiguration.
    \item \texttt{GET /zones/folders}: Liest verfügbare Ordner aus SharePoint und unterstützt damit die Auswahl eines Zielpfads.
    \item \texttt{POST /zones/\{zoneId\}/upload/preflight}: Übergibt eine Datei an den KI-Dienst und erzeugt einen temporären Upload mit erkannten Metadaten.
    \item \texttt{POST /zones/\{zoneId\}/upload/\{uploadId\}/commit}: Schreibt die bestätigte Datei nach SharePoint, persistiert Metadaten und stößt die Suchindex-Aktualisierung an.
    \item \texttt{POST /zones/sync-all}: Prüft SharePoint und Datenbank auf Konsistenz und entfernt verwaiste Dokumenteinträge.
    \item \texttt{GET /zones/documents/\{id\}/download}: Lädt eine Datei über die SharePoint-Referenz des Dokuments herunter.
    \item \texttt{GET /zones/documents/\{id\}/folder-link}: Liefert den Web-Link auf den übergeordneten SharePoint-Ordner des Dokuments.
    \item \texttt{GET /zones/documents/\{id\}/file-link}: Liefert den direkten Web-Link zur Datei in SharePoint.
\end{itemize}

Damit bündelt der Zone-Service die operative Hauptlogik der Anwendung in einer
zentralen API.

\subsubsection{Documents-Service}

Der Documents-Service verwaltet die persistenten Metadaten der Dokumente in der
PostgreSQL-Datenbank. Während die eigentlichen Dateien in SharePoint liegen,
werden hier strukturierte Informationen wie Dokumenttyp, Jahr, Partner oder
Eigentümer gespeichert.

\begin{itemize}
    \item \texttt{POST /documents}: Registriert ein neues Dokument nach dem Commit in SharePoint und speichert dessen Metadaten.
    \item \texttt{GET /documents/by-zone/\{zoneId\}}: Liefert alle Dokumentreferenzen einer Zone für einen bestimmten Benutzer.
    \item \texttt{DELETE /documents/\{id\}}: Löscht einen Dokumenteintrag aus der Datenbank.
    \item \texttt{GET /api/documents/\{id\}}: Gibt die Detaildaten eines einzelnen Dokuments zurück.
    \item \texttt{GET /api/documents/recent}: Liefert zuletzt erfasste Dokumente eines Benutzers, etwa für Dashboard und Erinnerungsansicht.
\end{itemize}

Der Service ist damit für die fachliche Verwaltung der Dokumentdaten zuständig,
während Dateiinhalt und Dateispeicherung bewusst ausgelagert bleiben.

\subsubsection{Search-Service}

Der Search-Service kapselt die Anbindung an Azure AI Search. Er stellt einerseits
die eigentliche Suchfunktion für das Frontend bereit und andererseits interne
Endpunkte zur Pflege des Suchindexes.

\begin{itemize}
    \item \texttt{POST /search}: Führt eine benutzerbezogene Suche mit Suchtext, Filtern und Facetten aus.
    \item \texttt{POST /internal/index}: Übernimmt ein Dokument in den Suchindex; dieser Endpunkt wird intern vom Zone-Service verwendet.
    \item \texttt{DELETE /internal/index/\{id\}}: Entfernt ein Dokument aus dem Suchindex, etwa wenn es aus dem System gelöscht oder als verwaist erkannt wurde.
\end{itemize}

Dadurch ist die Suche klar vom übrigen System getrennt und kann unabhängig
indiziert sowie abgefragt werden.

\subsubsection{AI-Service}

Der AI-Service übernimmt die automatische Dokumentklassifikation vor dem finalen
Upload. Er ist bewusst schlank gehalten und stellt derzeit im Wesentlichen einen
zentralen Fachendpunkt bereit.

\begin{itemize}
    \item \texttt{POST /ai/classify}: Nimmt eine PDF-Datei entgegen, analysiert sie mit dem KI-Modell und liefert erkannte Metadaten wie Dokumenttyp, Jahr, Partner oder Ablaufdatum zurück.
\end{itemize}

Dieser Endpunkt wird im Upload-Preflight verwendet, um dem Benutzer vor dem
finalen Commit bereits einen strukturierten Vorschlag anzuzeigen.

\subsubsection{Zusammenfassung der Implementierung}

Die API-Landschaft von DropIT ist damit serviceorientiert aufgeteilt:
\begin{itemize}
    \item Der Zone-Service steuert die fachlichen Hauptabläufe rund um Zonen, Upload und SharePoint.
    \item Der Documents-Service verwaltet die strukturierten Dokument-Metadaten.
    \item Der Search-Service stellt Suche und Indexpflege bereit.
    \item Der AI-Service übernimmt die automatische Voranalyse hochgeladener Dokumente.
\end{itemize}
